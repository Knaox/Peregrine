# Shops — multi-shop integration guide

Peregrine ships a native multi-shop architecture : any number of
third-party shops can sell your `ServerConfiguration` rows in parallel,
each with their own API key, their own outbound webhooks, and their own
scoped catalog access.

This page covers everything an operator or shop developer needs to know.

---

## 1. Mental model

```
┌─────────────────┐       ┌──────────────────┐       ┌────────────────┐
│ Peregrine admin │       │  Shop (third-    │       │ Customer       │
│ (you)           │       │  party billing)  │       │ (player)       │
│                 │       │                  │       │                │
│ • create Shops  │       │ • lists your     │       │ • picks a plan │
│ • mint API keys │       │   catalog        │       │ • pays Stripe  │
│ • create        │       │ • creates Stripe │       │ • gets a server│
│   ServerConfigs │       │   products       │       │                │
│ • attach via    │       │ • receives       │       │                │
│   pivot         │       │   webhooks       │       │                │
└─────────────────┘       └──────────────────┘       └────────────────┘
```

You never deal with prices, currencies, billing cycles or trials —
that's the shop's job. Peregrine is purely the technical surface :
"this configuration = X RAM, Y CPU, this Pelican egg, deploy here."

The bus that ties everything together is **Stripe**. Every transactional
event (checkout completed, subscription renewed, refund, dispute) flows
through `/api/stripe/webhook`. The shop tags every Stripe transaction
with four `peregrine_*` metadata keys so Peregrine knows which
configuration / shop / user / order is concerned.

## 2. Setting up a Shop (admin)

### A. Create the Shop row

Filament: `/admin/shops` → **Create**

| Field | Example | Notes |
|---|---|---|
| Name | `BlueSky Hosting` | Display name in Filament + Stripe metadata audit |
| Slug | `bluesky` | URL-safe stable identifier |
| Domain | `bluesky-hosting.com` | Informational only, no auth derived |
| Status | `active` | Suspending the shop blocks all inbound + outbound traffic |

### B. Generate an API key

On the shop's edit page → **Api Keys** relation manager → **Generate
API key** action :

| Field | Example |
|---|---|
| Label | `production-ci` |
| Abilities | `configurations:read`, `orders:read`, `webhooks:read`, `webhooks:write` |
| Expires at | (optional) |
| Env | `live` or `test` |

The plaintext token (`psk_live_a1b2c3d4...`) is shown **once** in a
modal — copy it immediately. Peregrine only stores its SHA-256 hash. If
you lose it, revoke + regenerate.

Send the token to the shop's developer through a secure channel
(Bitwarden, 1Password, encrypted email). Never put it in version
control.

### C. Attach `ServerConfiguration` rows

On the shop's edit page → **Server Configurations** relation manager →
**Attach** :

| Field | Example | Notes |
|---|---|---|
| Configuration | `mc-vanilla-medium` | Pick from your existing Filament list |
| Shop external ID | `plan-mc-medium` | Optional; the shop's own internal id |
| Visible | `true` | Hide from the public API without removing the link |
| Sort order | `0` | Order in `GET /api/v1/configurations` listings |

A configuration **not** attached to any shop is invisible to every
shop's API key — it's an admin-only template you can promote later.

## 4. Tagging Stripe transactions

Every Stripe Checkout Session / Subscription / PaymentIntent the shop
creates **must** carry these four metadata keys :

| Key | Type | Source |
|---|---|---|
| `peregrine_configuration_id` | int (string) | `id` from `GET /configurations` |
| `peregrine_shop_id` | int (string) | The shop's own `Shop.id` |
| `peregrine_user_email` | string | Buyer email |
| `peregrine_external_order_id` | string | Shop's own opaque order reference |

Optional :

| Key | Type | Use |
|---|---|---|
| `peregrine_server_id` + `is_resubscribe=true` | int + bool | Revives an existing `Server` instead of creating a new one |
| `peregrine_metadata` | JSON string | Free-form bag persisted to `Server.metadata` for audit |

The official PHP SDK provides a fluent builder :

```php
use Peregrine\ShopSdk\Stripe\MetadataBuilder;

$session = $stripe->checkout->sessions->create([
    'mode' => 'subscription',
    'line_items' => [[ 'price' => $priceId, 'quantity' => 1 ]],
    'success_url' => 'https://my-shop.example.com/success',
    'cancel_url'  => 'https://my-shop.example.com/cancel',
    'customer_email' => $buyer->email,
    'metadata' => MetadataBuilder::create()
        ->configuration(42)
        ->shop(7)
        ->user($buyer->email)
        ->order('shop-order-1234')
        ->build(),
]);
```

### What if the metadata is missing or invalid ?

Peregrine's `ResolveStripeMetadataAction` rejects with a structured
`skipped` reason :

| Reason | Cause |
|---|---|
| `missing_required_metadata` | One of the four keys is absent or empty |
| `unknown_shop` | `peregrine_shop_id` doesn't match a Shop row |
| `shop_suspended` | Shop status is `suspended` |
| `unknown_configuration` | `peregrine_configuration_id` doesn't match a row |
| `configuration_not_authorised_for_shop` | Pivot row is missing |

Peregrine **never** returns 4xx in this case — Stripe would retry the
event for hours. Instead the rejection is logged in
`stripe_processed_events`, an admin notification fires, and the
response is `200`. Check `/admin/bridge-sync-logs` to investigate.

## 5. Receiving outbound webhooks (shop side)

Whenever you edit a `ServerConfiguration` in Filament, Peregrine fans
out a signed POST to every active `WebhookEndpoint` of every Shop the
configuration is attached to (and that subscribed to the event type).

### Configuring an endpoint

Either via Filament (`/admin/webhook-endpoints` → **Create**) or via
the API (the shop key needs `webhooks:write`) :

```bash
curl -X POST -H "Authorization: Bearer psk_live_..." \
     -H "Content-Type: application/json" \
     -d '{
       "name": "Catalog sync",
       "url": "https://my-shop.example.com/webhooks/peregrine",
       "subscribed_events": ["configuration.created", "configuration.updated", "configuration.deleted"]
     }' \
     https://peregrine.example.com/api/v1/webhooks/endpoints
```

The 201 response carries `meta.signing_secret` **once**. Store it.

### Verifying the signature

Three headers ship with every delivery :

```
webhook-id        : <UUID>
webhook-timestamp : <unix seconds>
webhook-signature : v1,<base64 hmac-sha256>
```

Signed content is `{webhook-id}.{webhook-timestamp}.{raw_body}`.

Verification (PHP SDK) :

```php
use Peregrine\ShopSdk\Webhooks\StandardWebhookVerifier;

$ok = (new StandardWebhookVerifier())->verify(
    $request->header('webhook-id'),
    $request->header('webhook-timestamp'),
    $request->getContent(),
    $request->header('webhook-signature'),
    $endpointSigningSecret,
);
if (! $ok) abort(401);
```

Manual verification (any language) is documented in
[`/docs/standard-webhooks`](/docs/standard-webhooks).

### Event types

Catalog-only :

| Type | Trigger |
|---|---|
| `configuration.created` | Admin creates a `ServerConfiguration` |
| `configuration.updated` | Admin edits one |
| `configuration.deleted` | Admin deletes one |

No lifecycle events flow via Peregrine webhooks. Server provisioning
state is observed via Stripe events directly (the shop already gets
those) plus the `GET /api/v1/orders/{external_order_id}` poll endpoint.

### Retry policy

| Attempt | Delay |
|---|---|
| 1 | immediate |
| 2 | +60 s |
| 3 | +5 min |
| 4 | +15 min |
| 5 | +30 min |
| 6+ | +60 min (capped) |

After `endpoint.max_retries` exhaustion the delivery is marked
`expired` and surfaces in `/admin/webhook-deliveries`. Admin can replay
manually with one click.

## 6. The full lifecycle in one diagram

```
1. Shop creates Stripe Product + Checkout Session with metadata.peregrine_*
2. Customer pays
3. Stripe POSTs checkout.session.completed to /api/stripe/webhook
4. Peregrine validates metadata → resolves Shop + Configuration via pivot
5. Dispatch chain : LinkPelicanAccountJob → ProvisionServerJob
6. Pelican creates the server asynchronously
7. Server status flips: provisioning → active (via Pelican install webhook)
8. Customer gets a "server ready" email from Peregrine

(Throughout the lifecycle:)
- Stripe events drive subscription state (active, past_due, deleted, refund, dispute)
- Peregrine reacts: suspend, resume, terminate
- Shop can poll GET /api/v1/orders/{external_order_id} at any time

(When admin edits a configuration:)
- Observer fires → outbound webhook fans out to subscribed endpoints
```

## 7. Common pitfalls

| Pitfall | Fix |
|---|---|
| Token leaked in commit / log | Revoke immediately, regenerate, rotate signing secrets |
| Missing one of the four metadata keys | Use the SDK `MetadataBuilder` — it errors locally before hitting Stripe |
| Shop suspended → all checkouts silently rejected | Check `/admin/bridge-sync-logs` for `shop_suspended` reasons |
| Configuration not attached → 404 / `configuration_not_authorised_for_shop` | Verify the pivot in the shop's relation manager |
| Verifier accepts old timestamp | Hard-rule : `abs(now - timestamp) > 300` MUST reject |
| Verifier doesn't dedupe `webhook-id` | Add a per-shop replay ledger ; Peregrine never reuses an id but networks are flaky |
| Endpoint URL points at private IP | Peregrine refuses (DNS rebinding protection) — use a public URL |

## 9. SDK quick install

```bash
composer require peregrine/peregrine-shop-sdk
```

```php
use Peregrine\ShopSdk\Client;

$client = new Client('https://peregrine.example.com', $token);
$catalog = $client->configurations();
$me = $client->shopMe();
$order = $client->order('shop-order-1234');
$endpoint = $client->createWebhookEndpoint(
    name: 'Catalog sync',
    url: 'https://my-shop.example.com/webhooks/peregrine',
    subscribedEvents: ['configuration.created', 'configuration.updated', 'configuration.deleted'],
);
echo $endpoint['meta']['signing_secret']; // ONCE — store it
```

That's the whole integration. Stripe handles the money, Peregrine
handles the servers, and the shop ties them together via four metadata
keys + one Bearer token.
