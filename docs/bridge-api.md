# Peregrine Bridge API — for shop developers

This document describes the HTTP contract Peregrine exposes for **external shops**
(SaaSykit, Paymenter, custom Laravel/Node/Python billing platforms…) to push
plan definitions into a Peregrine instance.

The Peregrine admin holds the technical config (Pelican egg, node, docker
image, port mapping, runtime toggles). Your shop holds the business config
(name, price, RAM/CPU/disk promised to the customer). Bridge keeps these two
sides aligned without coupling either codebase to the other.

## Architecture

```
SHOP                            PEREGRINE                      STRIPE
  │                                 │                            │
  │  POST /api/bridge/plans/upsert  │                            │
  ├────────────────────────────────▶│  ← stores plan mirror      │
  │  HMAC-signed JSON payload       │    in `server_plans`       │
  │                                 │                            │
  │  DELETE /api/bridge/plans/{id}  │                            │
  ├────────────────────────────────▶│  ← deactivates plan        │
  │                                 │                            │
  │                                 │  POST /webhook/stripe (P3) │
  │                                 │◀───────────────────────────┤
```

The Bridge endpoints are **server-to-server** : your shop calls Peregrine
directly. No user authentication is involved — the call is authenticated by
an HMAC signature shared between the two systems.

Stripe-side provisioning happens through a separate channel (Stripe webhook
to Peregrine) — out of scope of this document.

## Client configuration

Your shop admin will typically expose a settings screen to configure the
Bridge client. **Configure only the base URL** of the Peregrine instance —
your client code appends the endpoint paths automatically.

✅ **Right** — configure a base URL :

```
https://peregrine.example.com
```

❌ **Wrong** — never include an endpoint path :

```
https://peregrine.example.com/api/bridge/plans/upsert
https://peregrine.example.com/api/bridge/
https://peregrine.example.com/api/bridge/plans/upsert/
```

Doing the wrong thing produces garbage URLs like
`https://peregrine.example.com/api/bridge/plans/upsert/api/bridge/ping` →
HTTP 404. A robust client implementation should either (a) validate and
reject any configured URL containing `/api/bridge/`, or (b) strip trailing
path segments before storing. Either behavior is preferable to silently
accepting a mis-configured URL.

## Authentication

Every Bridge request must carry two headers :

| Header | Format | Purpose |
|---|---|---|
| `X-Bridge-Signature` | `sha256=<hex digest>` | HMAC-SHA256 of the raw request body, using the shared secret |
| `X-Bridge-Timestamp` | `<unix milliseconds>` | Request time. Replay protection : Peregrine rejects timestamps more than 5 min off |

The shared secret is generated by the Peregrine admin in
**Admin → Bridge** (a new 64-char base64 secret is one click away). Copy
the value and configure it on your shop side.

### Signature algorithm

```
signature = "sha256=" + lowercase_hex( HMAC-SHA256( raw_body, shared_secret ) )
```

Use the **raw request body** (the exact bytes you'll send), not a re-serialized
version. Order of JSON keys must be stable between signing and sending.

### Verification timing

Peregrine compares signatures via `hash_equals()` (timing-safe). Your client
must do the same to avoid timing leaks if it ever validates Peregrine's
responses.

## Endpoints

### `POST /api/bridge/ping`

No-op health check. Use this from your shop's "Test connection" button to
verify the URL + shared secret are correct without creating any data on the
Peregrine side.

**Headers** : same as any Bridge call (signature + timestamp). Body should
be empty `{}` (sign the empty JSON `{}`).

**Response** (200 OK) :

```json
{
  "ok": true,
  "service": "peregrine-bridge",
  "version": "1.0",
  "received_at": "2026-04-22T18:00:00+00:00"
}
```

A 200 here means : URL reachable, signature valid, Bridge enabled. No DB
writes, no audit log entry. Safe to call as often as needed.

### `POST /api/bridge/plans/upsert`

Create a plan or refresh an existing one. Idempotent on `shop_plan_id` —
re-sending the same payload does **not** create duplicates.

**Headers** :
```
Content-Type: application/json
X-Bridge-Signature: sha256=<hex>
X-Bridge-Timestamp: <ms>
```

**Body** (JSON) :

```json
{
  "shop_plan_id": 42,
  "shop_plan_slug": "minecraft-4go",
  "shop_plan_type": "subscription",
  "name": "Minecraft 4Go",
  "description": "A 4GB Minecraft server with Paper and 5 plugins preinstalled",
  "is_active": true,

  "billing": {
    "price_cents": 700,
    "currency": "CHF",
    "interval": "month",
    "interval_count": 1,
    "has_trial": false,
    "trial_interval": null,
    "trial_interval_count": null,
    "stripe_price_id": "price_1QXxxxxx"
  },

  "pelican_specs": {
    "ram_mb": 4096,
    "swap_mb": 0,
    "disk_mb": 10240,
    "cpu_percent": 200,
    "io_weight": 500,
    "cpu_pinning": null
  },

  "checkout": {
    "custom_fields": [
      {
        "key": "server_name",
        "label": "Server name",
        "type": "text",
        "optional": false
      }
    ]
  }
}
```

**Field reference** :

| Field | Required | Type | Notes |
|---|---|---|---|
| `shop_plan_id` | yes | int | Unique stable identifier from your shop. Used as the upsert key |
| `shop_plan_slug` | yes | string ≤ 255 | Slug for display (e.g. URL fragment) |
| `shop_plan_type` | yes | enum | `subscription` or `one_time` |
| `name` | yes | string ≤ 255 | Customer-facing name |
| `description` | no | string | Free text description |
| `is_active` | yes | bool | When false → plan inactive in Peregrine, no new servers |
| `billing.price_cents` | yes | int | Price in cents (e.g. 700 for 7.00) |
| `billing.currency` | yes | string(3) | ISO 4217 (CHF, EUR, USD…) |
| `billing.interval` | required if subscription | enum | `day` / `week` / `month` / `year` |
| `billing.interval_count` | required if subscription | int ≥ 1 | Multiplier (1 month, 3 months…) |
| `billing.has_trial` | yes | bool | |
| `billing.trial_interval`, `trial_interval_count` | conditional | | When `has_trial` true |
| `billing.stripe_price_id` | no | string | Set when the shop has synced this plan to Stripe |
| `pelican_specs.ram_mb` | yes | int ≥ 128 | RAM in MB |
| `pelican_specs.swap_mb` | no (default 0) | int ≥ 0 | |
| `pelican_specs.disk_mb` | yes | int ≥ 256 | |
| `pelican_specs.cpu_percent` | yes | int ≥ 1 | 100 = 1 core, 200 = 2 cores… |
| `pelican_specs.io_weight` | no (default 500) | int 10-1000 | Pelican I/O priority |
| `pelican_specs.cpu_pinning` | no | string | `0,1,2` style pinning, or null |
| `checkout.custom_fields[]` | no, max 3 | array | Stripe Checkout native custom fields |
| `checkout.custom_fields[].key` | required | string ≤ 40 alphaDash | |
| `checkout.custom_fields[].label` | required | string ≤ 100 | |
| `checkout.custom_fields[].type` | required | enum | `text` / `numeric` / `dropdown` |
| `checkout.custom_fields[].optional` | required | bool | |

**Response** (200 OK) :

```json
{
  "peregrine_plan_id": 7,
  "synced_at": "2026-04-22T18:00:00+00:00",
  "status": "needs_admin_config"
}
```

`status` is either `ready` (egg + node configured by the Peregrine admin —
plan can provision servers) or `needs_admin_config` (admin must finish
configuring the technical side before any customer can buy).

### `DELETE /api/bridge/plans/{shop_plan_id}`

Soft-deactivate a plan (sets `is_active = false`). Servers already
provisioned from this plan keep their reference — only **new** purchases
are blocked.

**Headers** : same as upsert (signature + timestamp). Body is empty — sign
the empty string.

**Responses** :

| Status | Body | When |
|---|---|---|
| 200 | `{"deactivated_at":"<iso>"}` | Plan was found and deactivated |
| 404 | `{"error":"plan_not_found"}` | No plan with this `shop_plan_id` |

## Error responses

All error responses are JSON `{"error": "<key>"}`. The list of keys :

| HTTP | Key | Meaning |
|---|---|---|
| 401 | `bridge.invalid_signature` | Missing or wrong `X-Bridge-Signature` |
| 410 | `bridge.invalid_timestamp` | Missing or unparseable `X-Bridge-Timestamp` |
| 410 | `bridge.timestamp_expired` | Timestamp outside the 5-min replay window |
| 422 | (Laravel validation errors object) | Payload doesn't match the schema. Check the `errors` field for per-field details |
| 429 | (Laravel throttle) | Per-IP rate limit exceeded (60 req/min) |
| 503 | `bridge.disabled` | Bridge is OFF in the Peregrine admin |
| 503 | `bridge.secret_not_configured` | Shared secret hasn't been set in the Peregrine admin |

## Update semantics

When you re-push a plan (same `shop_plan_id`), Peregrine **only overwrites
shop-owned fields** (everything in the payload above). The technical config
the Peregrine admin set (egg, node, docker_image, port mapping, env vars,
runtime toggles, feature limits) is **preserved**. This means :

- Shop can change a plan's price freely — the existing server provisioning
  config stays intact.
- Peregrine admin can tune egg/docker/etc. without fear of being overwritten
  on the next sync.
- One-way ownership : every field has exactly one source of truth.

## Code samples

### PHP (Guzzle)

```php
use GuzzleHttp\Client;

$secret = 'your-shared-secret-from-peregrine-admin';
$payload = json_encode([
    'shop_plan_id' => 42,
    // … rest of the payload
], JSON_THROW_ON_ERROR);

$timestamp = (int) (microtime(true) * 1000);
$signature = 'sha256=' . hash_hmac('sha256', $payload, $secret);

$response = (new Client())->post('https://peregrine.example.com/api/bridge/plans/upsert', [
    'headers' => [
        'Content-Type'      => 'application/json',
        'X-Bridge-Signature' => $signature,
        'X-Bridge-Timestamp' => (string) $timestamp,
    ],
    'body' => $payload,
]);
```

### Node.js (fetch)

```javascript
import { createHmac } from 'node:crypto';

const secret = process.env.PEREGRINE_BRIDGE_SECRET;
const payload = JSON.stringify({
    shop_plan_id: 42,
    // … rest of the payload
});

const timestamp = Date.now();
const signature = 'sha256=' + createHmac('sha256', secret).update(payload).digest('hex');

const response = await fetch('https://peregrine.example.com/api/bridge/plans/upsert', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'X-Bridge-Signature': signature,
        'X-Bridge-Timestamp': String(timestamp),
    },
    body: payload,
});
```

### Python (requests)

```python
import hmac, hashlib, json, time, requests

secret = "your-shared-secret"
payload = json.dumps({
    "shop_plan_id": 42,
    # ... rest of the payload
}, separators=(',', ':'))

timestamp = int(time.time() * 1000)
signature = "sha256=" + hmac.new(
    secret.encode(), payload.encode(), hashlib.sha256
).hexdigest()

response = requests.post(
    "https://peregrine.example.com/api/bridge/plans/upsert",
    data=payload,
    headers={
        "Content-Type": "application/json",
        "X-Bridge-Signature": signature,
        "X-Bridge-Timestamp": str(timestamp),
    },
)
```

### cURL (smoke test)

```bash
SECRET='your-shared-secret'
PAYLOAD='{"shop_plan_id":1,"shop_plan_slug":"test","shop_plan_type":"subscription",...}'
TS=$(($(date +%s)*1000))
SIG="sha256=$(echo -n "$PAYLOAD" | openssl dgst -sha256 -hmac "$SECRET" | sed 's/^.* //')"

curl -X POST https://peregrine.example.com/api/bridge/plans/upsert \
  -H "Content-Type: application/json" \
  -H "X-Bridge-Signature: $SIG" \
  -H "X-Bridge-Timestamp: $TS" \
  -d "$PAYLOAD"
```

## Operational notes

- **Rate limit** : 60 requests/minute per source IP. Bulk syncs of large
  catalogs should batch + sleep accordingly.
- **Audit trail** : every Bridge call (success OR failure) is logged in
  the Peregrine admin under **Settings → Bridge sync logs**. Use this
  page to debug HMAC failures or payload validation issues.
- **Idempotency** : safe to retry an upsert with the same `shop_plan_id`
  any number of times.
- **Scrubbing** : if your `checkout.custom_fields[]` ever carries a `value`
  field (user-entered data captured at checkout), Peregrine strips it
  before persisting in the audit log. The schema doesn't accept a `value`
  field — this is for forward-compat protection.
- **Order of operations** : a plan can be pushed before being synced to
  Stripe (`billing.stripe_price_id` may be null). In that case, the plan
  exists in Peregrine but `status: needs_admin_config` until the Peregrine
  admin attaches an egg/node. Set the Stripe Price ID in a follow-up
  `upsert` once your shop has it.

## Stripe webhook (separate channel)

Provisioning a real game server when a customer pays goes through Stripe
webhooks pointing at `POST /api/stripe/webhook`. This is **independent**
from the plan-sync API above — Stripe sends events directly to Peregrine.

### Events Peregrine listens to

| Event | Outcome |
|---|---|
| `checkout.session.completed` | A new server is provisioned (Pelican createServer + local Server row) and the customer is emailed when ready. |
| `customer.subscription.updated` | If the price changed → server resources are upgraded/downgraded via Pelican `updateServerBuild`. If status went `past_due` → soft suspend. If status went back to `active` → unsuspend. |
| `customer.subscription.deleted` | Server is suspended immediately and scheduled for hard deletion after the grace period (admin-configurable, default 14 days). The customer receives an email with the deletion date. |
| `invoice.payment_failed` | Admin notification (informational). Stripe handles the dunning retry policy itself; the actual server suspension follows from the subsequent `subscription.updated → past_due`. |

### Configuration

1. **In Peregrine** : go to **Admin → Bridge** and enable the Bridge toggle.
2. **In Stripe Dashboard** : go to **Developers → Webhooks → Add endpoint**.
3. Endpoint URL : `https://{your_peregrine_domain}/api/stripe/webhook`
4. Events to enable : the 4 above (you can copy-paste `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`).
5. After creating the endpoint, click "Reveal signing secret" → copy the `whsec_…` value.
6. Back in **Peregrine → Admin → Bridge**, paste the secret in "Stripe webhook signing secret" → Save.
7. (Optional) Adjust **Grace period before hard delete** (default 14 days, range 0–90).

### Local development with Stripe CLI

```bash
stripe login
stripe listen --forward-to https://your-peregrine-dev.test/api/stripe/webhook
# → CLI prints a temporary whsec_… → paste in /admin/bridge-settings
stripe trigger checkout.session.completed
```

To trigger a flow end-to-end with a specific Price ID matching one of your
local `ServerPlan.stripe_price_id` :

```bash
stripe trigger checkout.session.completed \
  --override checkout_session:line_items[0][price]=price_YOUR_REAL_ID \
  --override checkout_session:metadata[server_name]=MyTestServer
```

### ⚠️ Queue worker required

The webhook handler **dispatches jobs** (provisioning, suspending, plan
upgrades) — it never calls Pelican synchronously, since Stripe gives 5
seconds max to respond. **A queue worker must be running** for the actual
work to happen :

```bash
php artisan queue:work --queue=default --tries=3 --max-time=3600
```

Without a worker, webhook calls succeed (Stripe sees 200) but the jobs pile
up in the `jobs` table and are never processed. See
[docs/operations/queue-worker.md](operations/queue-worker.md) for production
setup with supervisor / systemd.

### Idempotency

Stripe may retry an event for up to 3 days if a delivery fails (or
occasionally re-deliver after a 200). Peregrine maintains a
`stripe_processed_events` ledger keyed on `event.id` — duplicates short-
circuit with `200 {"received":true,"idempotent":true}` and zero side
effect. The ledger is auto-cleaned (`stripe:clean-processed-events`
artisan command, scheduled daily) of rows older than 30 days.
